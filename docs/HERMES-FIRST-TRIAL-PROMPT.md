# Hermes First Trial Prompt

这是一份可直接投喂给 Hermes 的首轮试验提示词。

目标不是让它“把 EAT 全自动做完”，而是验证它能否在当前仓库里完成一次有边界、有验证、有结论的自主开发闭环。

## 推荐用途

适合作为第一轮试验，用来验证：

- 它会不会先读对文档
- 它会不会把环境问题和代码问题分开
- 它会不会只改最小必要范围
- 它会不会主动跑验证命令

## 直接可用 Prompt

```text
你在 /home/code/EAT 工作。

先阅读：
1. AGENTS.md
2. README.md
3. docs/README.md
4. docs/HERMES-AUTONOMY-TRIAL.md

然后只继续阅读与当前任务直接相关的最小代码和文档集合，不要无差别扫描全仓。

任务目标：
对 EAT 做一轮“有边界的自主开发试验”。
你的目标不是重构整个系统，而是先验证当前仓库是否已经具备稳定的本地开发闭环；如果你发现一个明确、局部、可验证的问题，可以在最小范围内修复它。

硬约束：
- 不改任务状态机
- 不改 review / merge 语义
- 不改 API 路径，除非发现确定的实现缺陷且必须修复
- 不改部署脚本
- 不做大范围重构
- 不把 EAT 改造成无监督黑盒系统

工作方式：
1. 先用最小命令验证当前环境和仓库状态。
2. 如果是环境问题，明确指出，不要误报成代码回归。
3. 如果存在明确、局部、可验证的实现问题，只修最小必要范围。
4. 改动后运行与改动匹配的验证命令。
5. 最终给出清晰结论，不要泛泛而谈。

至少执行这些验证：
- docker image inspect eat/worker-base:latest
- cd /home/code/EAT/backend && go test ./...
- cd /home/code/EAT/web && pnpm lint && pnpm build

如果你做了跨前后端或任务创建相关改动，再执行：
- cd /home/code/EAT && npm test

完成后必须汇报：
- 你读了哪些文档
- 你检查了哪些代码
- 你是否发现真实问题
- 如果有改动，改了什么，为什么
- 跑了哪些命令
- 哪些通过，哪些失败
- 失败是代码回归还是环境前置条件问题
- 还有哪些风险需要人工确认

如果没有发现明确问题，不要为了“显得有产出”而编造修改；直接给出验证结论。
```

## 期望结果

理想情况下，Hermes 的首轮输出应该表现出这些特征：

- 它先读了根级文档，而不是直接乱改代码
- 它先验证环境，再判断代码
- 它只选择一个最小切口
- 它会在结尾区分“已验证通过”和“仍需人工确认”

如果它一开始就试图：

- 全仓大重构
- 擅自改产品定位
- 不跑验证
- 把环境失败说成代码缺陷

那这轮试验就可以判定为方法不稳定，而不是单纯“模型能力不够”。
