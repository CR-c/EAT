# Web 端黄金路径比对测试任务文档

> 日期：2026-03-27
> 范围：当前运行中的 EAT Web 页面
> 执行方式：真实页面点击 + 人工参与 leader 对话
> 约束：禁止对本项目执行任何 git 操作；若流程中需要提交代码，只允许在被注册的测试项目内操作

## 1. 目标

验证当前 Web 端是否能完成下面这条闭环，并将页面操作逻辑与业务实际能力逐步比对：

1. 注册项目
2. 创建任务
3. 与 leader 多轮对话，确认任务文档
4. 进入计划审阅
5. 测试批注打回并重新生成分配
6. 批准后进入执行
7. 查看文件差异
8. 测试让 leader 再次修改
9. 在结果无误后走到“可提交代码”的产品节点

本次执行不对本仓库做代码提交；若产品流程要求实际提交代码，只在单独创建的测试项目中验证，不在 `/home/code/EAT` 仓库中做 git 提交。

## 2. 测试原则

- 页面操作以当前实际 UI 为准，不按历史界面文档假设控件位置。
- 产品语义以 `docs/PRD.md`、`docs/phase/04-lead-session-chat-flow.md`、`docs/phase/05-plan-generation-validation-and-snapshots.md`、`docs/v1.1/17-web-leader-orchestration-and-team-lifecycle.md` 为准。
- 与 leader 的对话必须由人工完成，不依赖浏览器自动化脚本生成或决定对话内容。
- 除 leader 对话内容本身外，其余流程优先使用真实页面点击完成。
- 测试过程中发现问题时，先做局部修复，再从受影响步骤重新走流程。
- 若问题需要明显改动产品结构、状态机语义、执行模型或跨层重构，立即中止测试并记录阻塞。

## 3. 比对对象

### 3.1 业务目标链路

- 项目注册必须接受本地 Git 仓库并返回仓库状态。
- 任务创建必须生成可进入工作区的任务。
- 澄清阶段必须保留 leader 会话与消息历史，并通过显式确认进入规划。
- 计划阶段必须允许查看分配结果，并支持“无批注打回”与“带批注打回重拟”。
- 执行阶段必须能展示团队/节点执行态。
- 完成阶段必须能展示任务差异。
- 若产品提供最终修改入口，需验证能否通过 leader 触发并回到正确阶段。

### 3.2 当前页面映射

- 项目页：`/projects`
- 项目任务页：`/projects/:projectId/tasks`
- 创建任务页：`/projects/:projectId/tasks/new`
- 工作区页：`/projects/:projectId/workbench?taskId=...`

## 4. 环境与前置检查

执行前必须确认：

1. Go 后端可以正常启动，前端静态资源可被当前后端正确加载。
2. 当前环境具备 Web 端调用所需的 agent 运行条件。
3. 若执行阶段依赖 Docker sandbox，则 Docker 状态必须满足最小可运行要求。
4. 准备用于注册的独立测试 Git 仓库，避免触碰 `/home/code/EAT` 的 git 状态。
5. 明确测试记录方式：记录每一步的页面表现、接口报错、修复点与回归结果。

## 5. 测试数据准备

准备一个独立测试项目仓库，建议：

- 仓库规模小，便于多次生成与修改
- 含最小可运行前端或脚本入口，方便观察差异
- 允许在该测试仓库中做真实代码变更与提交验证

建议准备任务主题：

- 一个能拆出 architect / frontend / tester / integration 或相近角色的小型需求
- 需求应足以触发计划分配、执行、差异查看和收尾修改

## 6. 执行步骤

### 6.1 编写并复核测试输入

在正式点击页面前，先准备：

- 测试项目路径
- 任务标题
- 初始需求描述
- 至少两轮 leader 澄清时会补充的信息
- 至少一条计划批注
- 至少一条完成后要求 leader 修改的补充意见

### 6.2 注册项目

操作要求：

1. 从项目页进入“注册新项目”
2. 使用当前页面支持的路径输入或目录浏览方式选择测试仓库
3. 确认注册成功
4. 检查页面中项目名称、路径、默认分支、仓库脏状态是否符合实际

判定标准：

- 能成功注册合法 Git 仓库
- 非法路径或错误路径会给出可读错误
- 项目卡片中的基础仓库信息与真实仓库一致

### 6.3 创建任务

操作要求：

1. 从项目任务页进入“发布新任务”
2. 填写标题、描述、基线分支、任务分支、lead agent
3. 创建任务并成功进入工作区

判定标准：

- 任务创建成功且状态进入 `DRAFT` 或可直接进入澄清起点
- 工作区可正常加载
- 任务分支信息、基线信息与输入一致

### 6.4 与 leader 多轮对话并确认任务文档

操作要求：

1. 在工作区聊天区发出第一条消息启动澄清
2. 由人工继续与 leader 多轮对话，补充边界、验收标准、限制条件
3. 对照右侧任务文档区域，确认文档是否随对话反映需求
4. 在文档内容足够明确后，点击“确认文档，进入计划审阅”

判定标准：

- 消息能发送，leader 有可读回复
- 刷新页面后对话和文档内容仍存在
- 显式确认后任务不应停留在 `CLARIFYING`

特别说明：

- 此步骤的对话内容由执行者人工决定，不使用自动脚本生成
- 若 leader 回复明显偏离需求，应继续人工追问，而不是跳过

### 6.5 测试计划分配、批注打回与重新生成

操作要求：

1. 进入计划审阅后查看节点分配与依赖
2. 先执行一次“无批注打回”或“带着批注打回重拟”
3. 若使用批注打回，至少对一个节点填写明确修改意见
4. 等待系统重新生成计划
5. 比对新旧计划是否体现打回意图
6. 计划满足要求后批准执行

判定标准：

- 页面能展示计划节点
- 打回操作可以真实落地，不只是前端假按钮
- 重新生成后任务仍保持正确计划阶段语义
- 批准后任务进入执行阶段，而不是仅停留在计划草稿

### 6.6 执行与实现

操作要求：

1. 进入执行阶段后观察团队/节点视图
2. 检查执行日志、节点状态、阻塞/完成状态是否与业务语义一致
3. 等待实现完成或出现可处理问题

判定标准：

- 执行态页面可反映 lead 与 worker 的当前状态
- 执行失败时能看见足够的错误信息
- 若修复了阻塞问题，能重新执行并继续推进

### 6.7 完成后查看文件差异

操作要求：

1. 在完成态查看 diff 文件树与补丁内容
2. 检查 base/head 引用说明是否可读
3. 抽查关键文件差异是否与需求一致

判定标准：

- 能查看文件列表
- 能切换查看文件补丁
- 差异内容与实际产出相符

### 6.8 测试“让 leader 修改”

操作要求：

1. 在查看差异后，尝试通过当前产品允许的方式要求 leader 再次修改
2. 观察任务是否能回到正确的可修改阶段
3. 验证修改完成后差异是否发生预期变化

判定标准：

- 若产品提供该入口，应能完成一次“结果不满意 -> 要求 leader 修改 -> 再次生成/执行 -> 再次查看结果”的闭环
- 若当前产品未提供该入口，应明确记录为能力缺口，而不是主观推断为支持

### 6.9 到达“可提交代码”节点

操作要求：

1. 在结果无误后，确认产品是否给出“可提交/可收口”的明确节点
2. 若只在测试项目中允许验证提交，则仅在测试项目内验证，不碰 EAT 仓库 git
3. 若当前产品并未开放网页内最终提交动作，要记录实际停留点

判定标准：

- 结论必须区分“Web 端已实现提交能力”和“当前仅能达到完成/审查/差异确认阶段”

## 7. 问题处理策略

### 7.1 允许直接修复并重测的问题

- 明显的前端交互错误
- API 路由或参数不一致
- 页面状态未刷新
- 小范围运行时错误
- 文案导致操作误判但不涉及产品语义重构

### 7.2 需要中止测试的问题

- 必须改动核心状态机才能继续
- 必须重构 leader / worker 执行主链路才能继续
- 必须修改仓库级结构、持久化主模型或跨层架构才能继续
- 当前产品根本没有承载该步骤的后端能力，只是页面占位

## 8. 记录要求

测试过程中持续记录：

- 每一步的页面入口和实际按钮文案
- 成功/失败结果
- 失败时的错误表现
- 对应修复文件
- 修复后回归结果
- 是否与文档定义一致
- 是否只是“页面存在”而非“能力真实可用”

## 9. 最终输出要求

最终测试文档必须回答下面这些问题：

1. 是否能从 Web 端完成项目注册
2. 是否能从 Web 端完成任务创建
3. 是否能通过人工 leader 对话完成任务文档确认
4. 是否能真实完成计划打回与重新生成
5. 是否能进入执行并产出实际代码结果
6. 是否能查看最终差异
7. 是否能要求 leader 做最后修改
8. 是否能从 Web 端真正走到“提交代码”节点
9. 哪些问题已修复
10. 哪些问题属于结构性阻塞

## 10. 自审清单

执行前先确认本任务文档满足：

- 覆盖完整黄金路径
- 清楚区分人工对话与页面自动化边界
- 明确禁止对 `/home/code/EAT` 做 git 操作
- 明确了“先修复、再回归”的执行策略
- 明确了结构性阻塞的中止条件
- 最终结论可以直接被明早验收使用
